실제 개발에 바로 착수하실 수 있도록, 가장 최신 업데이트(데이터 구조 세분화 및 거래량 추가 등)를 모두 반영하여 하나의 마크다운 문서로 깔끔하게 정리했습니다.

📄 제품 요구사항 정의서 (PRD)
1. 프로젝트 개요
프로젝트 이름: stock log (스톡 로그)

한 줄 설명: 지정된 시간에 복수 종목의 현재가 및 거래량 데이터를 조회하여 구글 스프레드시트에 자동으로 기록하는 데이터 자동화 프로그램

유형: 데이터 수집 및 워크플로우 자동화 (Automation & Data Scraping)

난이도: 저~중급 (Python 스크립트 작성 및 GitHub Actions 스케줄러 활용)

2. 사용자 시나리오
누가 (Who): 여러 주식 및 가상자산 종목(약 30~50개)의 가격 변동과 거래량을 매일 추적하고 기록하고 싶은 개인 투자자

언제 (When): 매일 주식 시장 장 마감 직후 (예: 한국 시간 기준 오후 4시)

왜 (Why): * 매일 직접 현재가를 검색하고 시트에 옮겨 적는 단순 반복 작업을 자동화하여 시간을 절약하기 위함.

누적된 데이터를 활용해 자신만의 투자 수익률 추적 및 차트 추세 분석 레이아웃을 구축하기 위함.

3. 핵심 기능 목록
필수 기능 (Must-Have)
다중 종목 설정 읽기: 구글 시트에 등록된 복수의 종목 코드와 수집 여부(Y/N)를 프로그램이 실행 시점에 동적으로 읽어오는 기능

다중 종목 순회 수집: 읽어온 종목 리스트를 루프(Loop) 돌며 당일 종가, 등락률, 거래량 데이터를 순차적으로 수집하는 기능

자동 기록 (Data Append): 수집된 데이터를 구글 시트의 데이터 탭 가장 아래 행에 새로운 행(Row)으로 누적하여 추가하는 기능

예외 처리 및 에러 로그 기록: 특정 종목 수집 실패 시 전체 프로그램이 멈추지 않고 다음 종목으로 넘어가며, 에러 내역을 구글 시트 내 별도 에러 탭에 기록하는 기능

스케줄러 자동 실행: 사용자의 개입 없이 GitHub Actions의 Cron Job을 통해 매일 지정된 시간에 Python 스크립트를 자동 실행하는 기능

4. 기술 스택
언어: Python 3.x

실행 환경 및 스케줄러: GitHub Actions (ubuntu-latest 환경, Cron 스케줄러 활용)

데이터베이스 및 저장소: Google Sheets (Google Sheets API v4, gspread 라이브러리 연동)

데이터 수집 라이브러리: yfinance (해외 주식/가상자산) 또는 requests + BeautifulSoup / pykrx (국내 주식 크롤링)

보안 및 환경 변수: GitHub Secrets (구글 API 서비스 계정 JSON 키, 타깃 시트 URL 등 주요 보안 데이터 숨김 처리)

5. 화면 구성 및 데이터 구조
구글 스프레드시트 구조 (UI 및 DB 역할)
1) '설정(Setup)' 탭
사용자가 수집할 대상 종목들을 등록하고 관리하는 화면

ticker: 수집할 주식의 티커 또는 종목 코드 (예: AAPL, 005930)

종목명: 해당 종목의 이름 (예: 애플, 삼성전자)

수집여부(Y/N): Y로 설정된 종목만 프로그램이 순회하며 수집

비고: 특이사항 기재 (예: 미국 주식, 장기 적립식 종목)

2) '데이터로그(Data_Log)' 탭
매일 자동으로 주가 및 거래량 데이터가 누적되어 기록되는 핵심 화면

수집일자: 데이터가 수집된 날짜 (YYYY-MM-DD)

수집시간: 데이터가 실제 수집된 정확한 시간 (HH:MM:SS)

ticker: 종목 코드

종목명: 종목 이름

현재가: 당일 장 마감 시점의 종가 (숫자 형식)

등락률: 전일 대비 주가 변동 비율 (퍼센트 형식, 예: 1.5%, -0.8%)

거래량: 당일 총 거래량 (주 단위 숫자 형식)

상태: 데이터 수집 성공 여부 (정상 또는 예외 발생 시 오류)

3) '에러로그(Error_Log)' 탭
수집 과정에서 예외 처리가 발생했을 때 원인을 추적하기 위한 관리용 화면

발생일시: 에러가 발생한 타임스탬프 (YYYY-MM-DD HH:MM:SS)

ticker: 에러가 발생한 종목의 코드 (시스템 전체 에러일 경우 SYSTEM)

에러 내용: 예외 처리 구문에서 포착된 상세 에러 메시지

6. 상세 기능 명세
F-01: 실행 트리거 및 환경 구성
GitHub Actions의 workflow에 정의된 정해진 UTC 시간(한국 시간 장 마감 이후)에 따라 가상 인프라가 기동된다.

가상 환경 호스트 내에 Python 환경을 수립하고 requirements.txt에 명시된 라이브러리를 설치한다.

GitHub Secrets에 저장된 환경 변수를 호출하여 구글 크레덴셜(Credential) 인증을 수행한다.

F-02: 데이터 로드 및 다중 종목 순회 수집
인증 성공 후 지정된 구글 스프레드시트의 '설정' 탭을 활성화한다.

'수집여부' 컬럼의 값이 'Y'인 행의 데이터를 배열(Array) 형태로 로드한다.

수집 대상 종목 리스트(약 30~50개)를 순회하며 현재가, 등락률, 거래량 데이터를 요청한다.

[중요] 대상 금융 서버의 트래픽 차단(Rate Limit)을 방지하기 위해, 각 종목을 수집할 때마다 1~2초간의 딜레이(time.sleep)를 강제로 부여한다.

F-03: 데이터 적합성 검증 및 분기 기록
수집된 데이터가 정상적인 숫자 형식인지 검증한다.

정상 데이터: '데이터로그' 탭의 마지막 행(Last Row)을 탐색하여 상태를 정상으로 마킹한 뒤, 전체 데이터를 새로운 행으로 일괄 추가(Append)한다.

오류 데이터: 수집 중 에러(네트워크 타임아웃, 상장폐지된 티커 등) 발생 시 try-except 구문을 통해 예외를 포착한다. 해당 루프에서 데이터로그의 상태를 오류로 마킹하거나 패스하고, 즉시 '에러로그' 탭에 상세 원인을 기록한 뒤 다음 종목 수집을 이어간다.

7. 디자인 가이드
타임스탬프 표준화: 데이터베이스 정합성을 위해 수집일자는 YYYY-MM-DD, 수집시간은 HH:MM:SS 형태로 표준화하여 기록한다.

구글 시트 조건부 서식 적용: * '데이터로그' 탭의 '등락률' 컬럼에 구글 시트 조건부 서식을 적용한다.

등락률 > 0 (양수) : 붉은색 텍스트 (또는 초록색, 선호도에 따라 적용)

등락률 < 0 (음수) : 푸른색 텍스트 (또는 붉은색, 선호도에 따라 적용)

8. 제약사항 및 해결방안
GitHub Actions 실행 시간 오차: * 제약: GitHub Actions의 무료 크론(Cron) 스케줄러 특성상 서버 대기열 상황에 따라 지정된 시간보다 1~5분 정도 지연 실행될 수 있다.

해결방안: 실행 시점이 아닌 데이터 수집 시점의 시스템 현재 시간을 정확하게 파악하여 타임스탬프(수집시간)로 남김으로써 데이터 기록의 신뢰성을 보장한다.

API 호출 한계 및 크롤링 차단 제약: * 제약: 30~50개의 종목에 대해 연속적으로 웹 스크래핑 및 API 호출을 시도할 경우 IP가 임시 차단될 위험이 있다.

해결방안: 루프(Loop) 내부 코드에 지연 시간(time.sleep(1))을 보장하여 사람의 일반적인 조회 속도와 유사하도록 동작 메커니즘을 구성한다. 전체 실행 시간은 1~2분 내외로 GitHub 무료 제공량 내에서 안전하게 구동된다.
